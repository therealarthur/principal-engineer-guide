The Principal Engineer's AI Playbook

Everything a software engineer needs to think, speak, and build like a principal-level AI systems engineer. Organized from most important to most detailed.

🧠 The Mindset

A principal engineer is not the person who knows the most trivia. They are the person who can structure ambiguity, identify the critical decisions, and protect a system from predictable failure.

The 5 Mental Shifts

1. Reduce the problem first Most Important

Before solutions, before tools, before architecture: understand the actual problem. What matters most? What is the smallest useful version? What can wait?

Junior engineers hear a problem and start coding. Staff engineers hear a problem and start asking questions.

2. Think in tradeoffs, not absolutes

Nothing is universally "good" or "bad." Microservices are not inherently better than monoliths. RAG is not always better than fine-tuning. The question is always: for this problem, at this stage, with these constraints, which tradeoff is right?

3. Separate reversible from irreversible decisions

Swapping a UI library is easy. Changing a tenancy model after launch is painful. Principal engineers spend their energy on the hard-to-reverse decisions and move quickly on everything else.

Easy to reverse

Swap a library, change a UI component, adjust log format, rename a variable, add a feature flag

Hard to reverse

Schema design, auth model, tenancy boundary, API contracts published externally, encryption strategy

4. Think in failure modes

For every system, ask: How can this break? How will we know? How do we recover?

This separates someone who can build a demo from someone who can build a product.

5. Use AI to accelerate, not replace judgment

AI is a lever. It can brainstorm options, draft docs, generate scaffolding, surface risks. But you still own: prioritization, tradeoff selection, risk tolerance, and the final architecture.

Think of a principal engineer like an experienced general contractor. They do not personally wire every circuit or lay every tile. But they know which walls are load-bearing, which decisions cascade into expensive rework, and when the electrician is doing something that violates code. They spend most of their time on sequencing, risk, and integration, not on hammering.

Whenever you approach any engineering problem, use these 5 questions in order:

1. What are we building?

2. How should it work?
→
3. How do we keep it safe?
→
4. How do we ship it?

  2. How should it work? Requirements, architecture, data flow, interfaces

If someone asks you "How do you break down a problem?" in an interview, this 5-step framework is a clean, senior-sounding answer every time.

How Principal Engineers Filter What Matters

When faced with 50 possible concerns, they filter by:

  • Impact — Does this affect users, security, correctness, or revenue?
  • Risk — Could this lose data, expose secrets, or block scale?
  • Reversibility — Can we change it later cheaply?
  • Frequency — Will this happen often enough to justify design attention?

If something is high-impact, hard to reverse, and happens often: that is where you spend your engineering budget.

🎯 Problem & Requirements

Everything starts here. The biggest engineering failures are not code bugs. They are building the wrong thing, or building the right thing without understanding what "done" means.

Step 1: Define the Problem

Before any technology, answer:

  • Who is the user?
  • What pain are we solving?
  • What does success look like, measured?
  • What is MVP vs later?
  • What should the system not do yet?
The #1 failure mode at enterprise companies: scope creep before V1 ships. Define non-goals explicitly. Write them down.

Step 2: Functional vs Non-Functional Requirements

Functional Requirements (FR)

What the system does.

  • User can upload documents
  • System extracts invoice fields
  • Admin can view audit log
  • API returns proposal draft
Non-Functional Requirements (NFR)

How well the system must behave.

  • Response under 2 seconds
  • 99.9% uptime
  • Data encrypted at rest
  • System is auditable
Functional requirements are the menu at a restaurant: what dishes you serve. Non-functional requirements are the health code, the wait time, and whether the kitchen can handle a Friday rush.

NFR categories worth memorizing

Scalability

Reliability
How often does it break?
Auditability
Can we prove what happened?
Maintainability
Can someone else work on this?
Security
Is it protected?
Compliance
Does it meet regulatory rules?
Cost
Can we afford to run it?

🏗 Architecture

Architecture is the set of decisions that are expensive to change later. Everything else is implementation detail.

The Fundamental Choice: System Shape

Monolith

One big box. Everything together.

MVP / small team
Modular Monolith

One deploy, clean internal walls.

Sweet spot
Microservices

Separate boxes, separate deploys.

Large teams
The principal-level answer: "For an enterprise MVP, I'd lean toward a modular monolith with clean internal boundaries. This lets us ship fast without premature complexity, while keeping the option to extract services later based on actual, not hypothetical, scaling needs."

Key Architecture Decisions

These are the decisions principal engineers lose sleep over:

Data: source of truth

Where does authoritative data live? Who owns it? How do other parts access it? Getting this wrong early creates cascading pain.

Interfaces: API contracts

How do services and components communicate? Define contracts before implementation. Build against interfaces, not internals.

Sync vs Async

Sync (request/response): Simple, immediate. REST APIs, RPC calls.

Async (event-driven): Resilient, decoupled. Message queues, event streams.

Most systems use a mix. The decision is: which operations need immediate response, and which can tolerate delay?

Where AI lives vs where deterministic logic lives

This is critical for AI systems. Use the LLM for language, ambiguity, summarization, extraction. Keep deterministic code for permissions, pricing, calculations, policy enforcement, validation.

Never let the LLM become the policy engine. If the model decides who has access, what the price is, or whether an action is approved, you have built a system that cannot be audited, tested, or trusted.

🔒 Security

Security is not a feature you bolt on at the end. It is a property of how the system is designed from the start.

The 5 Security Layers

Think of security as 5 concentric rings, from outer to inner:

Layer 1 Identity & Access

Who are you, and what are you allowed to do?

Authentication (AuthN)
Proving who you are. "Show me your badge."
Authorization (AuthZ)
What you're allowed to do. "Your badge gets you into rooms 1-3, not room 4."
RBAC
Role-Based Access Control. Permissions tied to roles (admin, editor, viewer).
Least Privilege
Give the minimum access needed. No more.
MFA
Multi-Factor Authentication. Badge + fingerprint, not just badge.
SSO
Single Sign-On. One login for many systems.
Layer 2 Data Security

Is data protected while stored and while moving?

TLS / HTTPS
Encrypts data while it travels over the network. Like a sealed envelope vs. a postcard.
Encryption at rest
Data encrypted while stored (DB, disk, backups).
Secrets management
API keys, passwords, tokens stored in a vault, not in code.
Sanitization
Cleaning dangerous input. Removing harmful HTML, SQL, or script injection.
Redaction
Masking sensitive data: SSN becomes ***-**-1234.
Data classification
Labeling data as public, internal, confidential, or regulated.
PII
Personally Identifiable Information. Names, emails, SSNs, addresses.
Encryption at rest is a locked safe. Encryption in transit is an armored truck. You often need both.
When exactly do you use encryption?

Encryption in transit (TLS/HTTPS): Almost always. Between browser and server, between services, to databases, to third-party APIs.

Encryption at rest: Customer data, credentials, backups, audit logs, stored documents, vector stores if they contain sensitive content. For enterprise/government work, this is usually non-negotiable.

Layer 3 Application Security
Input validation
Never trust user input. Check, sanitize, constrain.
Output encoding
Prevent XSS by encoding output before rendering.
Rate limiting
Cap requests to prevent abuse or cost explosions.
Dependency scanning
Check libraries for known vulnerabilities.
SBOM
Software Bill of Materials. Inventory of everything in your software.
Layer 4 Infrastructure Security
Segmentation
Separate workloads by trust level. Dev, staging, prod are different zones.
Egress control
Control what can leave your network.
Image scanning
Check container images for vulnerabilities before deployment.
Zero Trust
Don't assume "inside the network" = trusted. Verify everything.
Layer 5 AI-Specific Security

This is the new frontier that most engineers ignore. If you know this, you stand out.

Prompt injection
Malicious instructions hidden in user input or documents that trick the model.
Data poisoning
Bad training data that corrupts model behavior.
Model tampering
Compromised model weights or artifacts.
Retrieval poisoning
Malicious content in retrieved documents manipulates AI output.
Data exfiltration
AI leaking sensitive info it should not share.
Tool allowlists
Model can only call pre-approved tools.
Human-in-the-loop
Human reviews/approves before high-risk actions.

Protecting a Trained Model from Corruption

There are 3 different timeframes where a model can be "corrupted":

During Training: Poisoning / Tampering

Bad data gets into training, or the model artifact itself is modified.

Protections: trusted data sources, dataset versioning, signed model artifacts, restricted model registry access, hash verification, isolated training environments.

After Deployment: Drift

The world changes but the model stays the same. Performance degrades silently.

Protections: regular evals, baseline comparisons, shadow testing, production monitoring, retraining cadence.

During Use: Prompt Injection / Retrieval Poisoning

The model is fine, but runtime inputs manipulate its behavior.

Protections: treat content as untrusted, separate instructions from data, validate outputs, restrict tool access, log interactions.

"If you mean corruption of a trained model, I'd break that into training-data poisoning, model artifact tampering, pipeline compromise, and runtime manipulation like prompt injection or poisoned retrieval. My approach would be dataset provenance, signed versioned artifacts, restricted promotion paths, validation before deployment, and continuous monitoring for drift or unexpected behavior."

Umbrella Security Strategies

These are the big strategic frameworks that tie the layers together:

  • Secure by Design — Build security in from the start, not bolted on later.
  • Zero Trust — Never assume internal = trusted. Continuously verify identity, device, and context.
  • Defense in Depth — Multiple layers of controls so no single failure exposes everything.
  • Least Privilege — Minimum permissions needed, nothing more.
  • NIST AI RMF — Risk management framework for trustworthy AI systems.
  • OWASP LLM Top 10 — The most common LLM-specific security risks (prompt injection, data poisoning, insecure output handling, etc.).

📊 Data & Machine Learning

ML is not magic. It is: use historical data to learn a pattern that helps predict, classify, or estimate something on new data.

ML vs LLM: The Core Difference

Machine Learning (ML)

The broad field: using data to train models that make predictions, classifications, or decisions.

Examples: spam detection, churn prediction, revenue forecast, fraud detection, lead scoring.

Large Language Model (LLM)

A specific type of AI model trained on massive text data to understand and generate language.

Examples: document Q&A, proposal drafting, text summarization, invoice extraction.

Not everything is an LLM problem. Revenue forecasting? That is probably classic ML. Document understanding? That is probably an LLM task. The best systems often combine both.

The 4 ML Problem Types

Classification Most common in enterprise

Predict a category. Spam or not spam? Fraud or not fraud? Which department owns this ticket?

Regression

Predict a number. Expected revenue? Home price? Delivery time? Risk score?

Clustering

Find groups without predefined labels. Segment customers by behavior. Group similar incidents.

Forecasting

Predict the future based on time-ordered data. Next quarter's revenue. Tomorrow's traffic. Monthly demand.

The ML Process: A Restaurant Analogy

Imagine you are opening a restaurant and want to predict how many customers will come each day so you can prep the right amount of food.

  • The target (label) is: how many customers tomorrow
  • One training example is: one past day, with all its context
  • Features are: day of week, weather, holiday?, nearby events, last week's count, season
  • Training: the model studies hundreds of past days and learns patterns
  • Evaluation: you test on recent days the model hasn't seen
  • Leakage: if you accidentally include "actual customers that day" as a feature, the model cheats

The Real ML Process

Step by step:

1. Define the task

What exactly are we predicting? "Will this customer churn?" or "What will revenue be next quarter?"

This is called defining the target or label. Bad label definition = bad model, no matter how fancy the algorithm.

2. Define one training example

What is one "row" in your dataset? One customer? One transaction? One month? One document?

This sounds trivial but getting it wrong is surprisingly common and catastrophic.

3. Feature engineering This is where value lives

Features are the input variables the model uses to learn. Raw data is rarely useful as-is. The art is creating meaningful features from raw signals.

Examples from raw order data:

  • Number of orders in the last 30 days
  • Average time between orders
  • Days since last order
  • 3-order rolling average of spend
  • Whether purchases spike on weekends
  • Ratio of current order to prior average

This is exactly the "connecting things 2 or 3 orders apart" idea. It is a real, important concept called temporal feature engineering.

4. Train/validation/test split

Never test the model on data it trained on. Split into:

  • Training set: model learns from this
  • Validation set: used to tune or compare models
  • Test set: final honest check on unseen data

For time-based data, splits must be chronological: train on the past, test on the future. Random splitting leaks future information.

5. Start with a baseline

Always start simple. Logistic regression, decision tree, or even "predict the average." If your fancy model barely beats the simple one, the complexity may not be worth it.

Common model families for tabular/business data: Logistic regression, decision trees, random forest, XGBoost/LightGBM (gradient boosting). These are workhorses of enterprise ML.

6. Evaluate properly

Metrics depend on the task:

Accuracy
How often the model is right overall
Precision
Of things the model flagged, how many were correct?
Recall
Of things that actually mattered, how many did the model catch?
MAE / RMSE
How far off a number prediction is, on average

The Data Pitfalls

Leakage Silent killer

The model accidentally sees information it wouldn't have at prediction time. Example: predicting churn but including "account closed date" as a feature. The model looks brilliant in testing but is useless in production.

Imbalanced classes

Only 1% of transactions are fraud, 99% are normal. "Predict normal every time" gets 99% accuracy but catches zero fraud. You need class-aware metrics like precision/recall.

Drift

Data patterns change over time. A model trained on 2023 data may quietly degrade in 2026 because customer behavior shifted.

Bias

The training data reflects historical unfairness, and the model learns to perpetuate it.

"First I'd define the business problem as a concrete prediction, classification, or forecasting task. Then I'd identify what one training example should be and define the target label clearly. I'd gather and clean relevant historical data, join related sources, and engineer useful features, especially temporal or behavioral features if time and sequence matter. I'd split the data, start with strong baselines, and evaluate with metrics appropriate to the problem, while checking for leakage, bias, and drift before thinking about deployment."
Feature engineering terms to know
Lag feature
Previous value(s). Last month's sales.
Rolling window
Average/sum over recent range. 7-day avg.
Temporal feature
Time-based: hour, day of week, month.
Entity join
Connecting data from different tables.
Signal vs noise
Useful pattern vs random variation.
Aggregation
Count, sum, avg, min, max by group.

🤖 AI / LLM Systems

LLMs are powerful but they are tools, not solutions. The system around the model is usually more important than the model itself.

The 3 Ways to Make an LLM Useful

1. Prompting Start here

Give the model instructions and examples. No training, no external data lookup.

Good for: summarization, rewriting, classification of provided text, extraction from short inputs.

Limitation: The model only knows what's in its training data and what you put in the prompt.

2. RAG (Retrieval-Augmented Generation) Most enterprise use cases

Before generating an answer, the system retrieves relevant documents and adds them to the prompt. The model answers from those documents.

Good for: internal knowledge bases, policy Q&A, proposal drafting, documentation search.

Why: freshness, traceability, citations, no hallucination from stale training data.

3. Fine-tuning

Train the model on company-specific examples to change its behavior or style.

Good for: consistent output format, domain-specific behavior patterns, tasks where prompting alone is not reliable enough.

Not good for: giving the model access to changing documents. That's RAG, not fine-tuning.

📂

RAG

Give the model a filing cabinet. It looks things up every time.

  • + Knowledge stays fresh
  • + Can cite exact sources
  • + Update docs, not the model
  • + Traceable answers
  • - Retrieval can fail
  • - Adds latency per request
Best for
Internal docs, policy Q&A, proposal drafting, anything where knowledge changes
VS
🧠

Fine-tuning

Send the model to training. It internalizes new behavior.

  • + Consistent style/format
  • + Faster at inference
  • + Learns behavior patterns
  • + Can handle niche tasks
  • - Knowledge gets stale
  • - Expensive to retrain
Best for
Consistent output format, company writing style, specialized repeated tasks
The most important distinction: RAG = knowledge access. Fine-tuning = behavior change. If the model needs to know your latest policies, use RAG. If the model needs to consistently write in your company's tone, consider fine-tuning.

RAG: The Full Pipeline

RAG is like an open-book exam. Instead of memorizing everything, the student (model) gets to look up relevant pages (retrieved chunks) before answering. The quality depends both on finding the right pages AND reading them correctly.

Key RAG concepts

Embedding
Numeric vector that represents the meaning of text. Similar meanings = close vectors.
Vector store
Database that stores embeddings and searches by similarity. Examples: FAISS, Pinecone, pgvector, Weaviate.
Chunking
Splitting docs into pieces. Too small = lost context. Too large = noisy retrieval.
Top-k retrieval
Return the k most relevant chunks.
Reranking
Second-pass sorting for better relevance.
Metadata
Tags like source, date, author, ACL attached to each chunk.
Hybrid search
Combine semantic (meaning) search with keyword search. Often better than either alone.
Grounding
Forcing the model to answer from provided sources, not its imagination.
Faithfulness
Is the answer actually supported by the retrieved sources?
Relevance
Did we retrieve the right chunks for this question?

RAG failure modes

  • Bad chunking: answer-relevant info split across chunks
  • Stale documents: policies changed but index was not refreshed
  • Weak embeddings: similar meanings are not close in vector space
  • Irrelevant top-k: the "most similar" chunks are not actually useful
  • Context overload: stuffing too much into the prompt hurts quality
  • Prompt injection via retrieved content
  • Missing permission filtering: users see docs they shouldn't

Agents vs Workflows

Agent

The model dynamically chooses what to do next. Multi-step reasoning, tool selection, branching, stateful progress toward a goal.

Use when: The path is not known in advance. Requires judgment, dynamic tool use, or complex branching.

Risks: Hard to test. Unpredictable. Can loop, misuse tools, or take wrong paths.

Deterministic Workflow

Predefined steps in a fixed pipeline. Each step is explicit. LLM may power individual steps but doesn't control the flow.

Use when: The path is known. Steps are predictable. Easier to test, audit, and secure.

When to prefer: Almost always start here. Use agents only when workflows genuinely cannot handle the problem.

"I'd only use agentic behavior when the problem truly requires dynamic tool selection or multi-step reasoning where the path is not known in advance. If the workflow is predictable and fixed, I'd rather build a deterministic pipeline because it's easier to test, cheaper, and safer."

Agent safety controls

  • Tool allowlists: model can only call approved tools
  • Bounded loops: max iterations/retries to prevent runaway
  • Action validation: tool calls checked against policy before execution
  • Approval gates: human approval before irreversible actions
  • Deterministic enforcement: permissions, pricing, policy in code, not in the model

5 Real-World Scenarios

Remember these scenarios and the terms naturally attach to each one:

Scenario 1: Internal Document Chatbot

"What does our PTO policy say?" The LLM searches company docs and answers from them.

Pattern: RAG. Key terms: retrieval, embeddings, vector DB, chunking, grounding, citations, hallucination prevention.

Scenario 2: Invoice / Contract Extraction

Read messy documents and pull out structured fields: vendor name, amount, date.

🚀 Delivery & Operations

Shipping is not the end. Operating is part of the design. A system that works on your laptop but cannot be safely deployed, monitored, or rolled back is not production-ready.

CI/CD: The Clean Explanation

Think of a factory assembly line. CI is the quality inspection station: every piece gets checked automatically before it moves forward. CD is the packaging and delivery truck: once inspection passes, the product gets shipped reliably to the store shelves.
CI: Continuous Integration

When code is pushed/merged:

  • Build
  • Lint & type-check
  • Run unit tests
  • Run integration tests
  • Security scan (optional)

Goal: Catch issues early, keep main branch healthy.

CD: Continuous Delivery/Deployment

After CI passes:

  • Package artifact/container image
  • Promote to environment (staging, prod)
  • Deploy with health checks
  • Verify deployment health
  • Rollback if needed

Goal: Ship safely and repeatedly.

Deployment Strategies

Blue-Green

Two identical environments. Blue is live, green gets the new version. Switch traffic when green is verified. Instant rollback: just switch back to blue.

Tradeoff: Simple rollback, but costs double the infrastructure.

Canary

Route a small percentage of traffic (e.g. 5%) to the new version. Monitor. If healthy, gradually increase. If not, roll back.

Tradeoff: Lower risk than big-bang deploys, but more complex routing.

Rolling

Replace instances one at a time. Old and new run simultaneously during the rollout.

Tradeoff: Simple, but rollback is slower. Default in most Kubernetes setups.

"Each feature branch or AI-assisted workstream would be isolated. Changes go through automated validation: unit tests, linting, type checks, and integration tests. On merge to main, CI builds and packages versioned container images. A CD system like Argo CD would reconcile the target Kubernetes environment to the desired state in Git. For production, I'd want health checks, staged rollout, and a clear rollback strategy."

The 4 Operational Signals

After deployment, you need to see what is happening:

Metrics

Numbers over time. Request rate, error rate, latency, CPU, queue depth, business counters.

Tool: Prometheus (collects and stores time-series metrics).

Logs

Discrete events and messages. Errors, warnings, informational records.

Tools: Loki, Elasticsearch, CloudWatch Logs.

Traces

The path of a single request across multiple services. "This request hit the API, then the auth service, then the database, then the AI model."

Tools: Jaeger, Tempo, Datadog APM.

Audit trail Different from logs

Structured, durable record of who did what, when, and under what authority. Not debug output. Business-critical accountability.

For AI systems, includes: prompt version, model version, retrieved doc IDs, tool calls, validation results, approval events.

Metrics tell you something is wrong. Logs tell you what happened. Traces tell you where it happened. Audit trails prove who did what. These are different concerns.

The Observability Stack

How these tools relate to each other:

OpenTelemetry The instrumentation layer

A vendor-neutral standard for instrumenting your code to emit metrics, logs, and traces. You add it to your services, and it exports data to whatever backend you choose. This is probably what you're thinking of as "the thing that hooks into all your functions."

Prometheus + Grafana Open source stack

Prometheus: Collects and stores metrics. Supports alerting rules. Not an audit system or log store.

Grafana: Visualization layer. Builds dashboards from metrics, logs, and traces. The "pane of glass" that sits on top.

Datadog Managed commercial platform

Metrics + logs + traces + APM in one managed platform. Faster setup, less ops. Also has its own audit trail feature for platform activity.

"I'd separate metrics, logs, traces, and audit events instead of pretending they're all the same thing. For instrumentation, I'd use OpenTelemetry so context propagates across services. For metrics, Prometheus is strong in Kubernetes environments, with Grafana for dashboards. For audit trails specifically, I'd emit structured business and security events to a retention-controlled system rather than relying on ad hoc application logs."

🏢 Building a Company LLM

"Make it our company's AI" is actually 4 different layers. Most teams only do layer 1, which is why their product feels like a ChatGPT wrapper.

The 4 Layers That Make It "Yours"

Layer 1 Company Knowledge (RAG)

Connect the model to your internal documents: policies, proposals, standards, architecture docs, naming conventions. The system retrieves these at runtime so the model answers from your actual data, not generic training.

This is the single biggest lever.

Layer 2 Company Behavior (Workflow & Policy)

Define rules: always cite sources. Never speculate on legal matters. Escalate HR questions. Require approval before risky actions. Use structured output for proposals. Enforce internal style guides.

This is built with: system prompts, tool policies, validators, orchestration logic, approval gates.

Layer 3 Company Style (Fine-tuning)

If prompting and RAG are not enough for consistency, fine-tune on curated internal examples. Instruction/response pairs, preferred outputs, reviewer feedback data.

Only do this after you know which tasks need it and can measure improvement.

Layer 4 Company Control (Self-hosting)

Host the inference, embeddings, vector stores, and logs on your own infrastructure. Data stays inside your boundary. Full operational control.

Think of it like onboarding a new employee. Layer 1 is giving them the company handbook and project files (knowledge). Layer 2 is teaching them the rules: when to escalate, how to format deliverables, what needs approval (behavior). Layer 3 is mentoring them until they match the team's communication style (tone). Layer 4 is where they sit: in your office, on your network, using your equipment (control).

The Maturity Ladder

Level 1: Wrapper Shallow

"System prompt says: you are the Company X assistant."

Gives tone. No real company capability. This is what most people ship and it's why it feels like a gimmick.

Level 2: Wrapper + RAG Real but basic

System prompt + retrieval over internal docs. Now it can actually answer from your company's data. First real enterprise version.

Level 3: Internal Platform Production-grade

RAG + company-specific prompts + tool access + structured workflows + citation requirements + approval boundaries + audit logging. Feels like a real internal company AI system.

Level 4: Tuned Platform Advanced

Level 3 + fine-tuning on internal examples + specialized extraction/drafting + model routing + internal hosting + domain-specific evaluation.

Self-Hosting: The Technical Stack

If data control matters, this is what the actual stack looks like:

Base model

Open-weight model you are licensed to self-host (e.g. Llama, Mistral).

Inference server

vLLM: Open-source, provides an OpenAI-compatible API so your apps don't need to change their integration.

NVIDIA NIM: Enterprise inference microservices for GPU hosting.

Retrieval layer

Internal embedding model + vector store (pgvector, Weaviate, Milvus) + ingestion pipeline + ACL-aware retrieval.

Gateway / policy layer

Your application enforces: auth, audit, rate limiting, prompt templates, tool permissions, output validation.

Infrastructure

GPUs, storage, secrets management, networking, observability, backups, CI/CD, access controls.

Fine-Tuning Data: What Shape?

You need two kinds of data, and they are different:

Knowledge data (for RAG)

Documents the system searches at runtime:

  • Policies, standards, architecture docs
  • Historical proposals and deliverables
  • Internal glossaries and procedures
Behavior data (for fine-tuning)

Instruction/output pairs that teach behavior:

  • Ideal internal answers to common questions
  • Good vs. bad draft comparisons
  • Escalation and refusal examples
  • Style/format demonstrations
Documents teach the system what your company knows. Instruction/output pairs teach the system how your company behaves. These are not the same.

Air-Gapped / Disconnected Strategies

True air gap

No network connection to outside. Manual transfer of models, data, packages. Offline vulnerability scanning. Human-reviewed import of every artifact.

This is the strictest form, typically government/classified environments.

Semi-isolated enclave

Controlled ingress/egress. Internal artifact mirrors (PyPI, Docker, npm). Model artifacts approved before import. Admin access tightly controlled.

More practical for most enterprises.

Internal CI artifact mirror: a private copy of build dependencies (Docker images, Python packages, npm packages, approved models) so builds never pull directly from the public internet. Reduces supply-chain risk, improves reproducibility, and supports restricted environments.

"To make an LLM genuinely company-aware, I wouldn't start by training from scratch. I'd start with a strong base model, ground it in internal data using RAG, put it behind a policy and orchestration layer that enforces citations, permissions, and approval boundaries. If specific tasks still needed more consistent behavior, I'd add supervised tuning on curated internal examples. For hosting, if data control was important, I'd self-host inference with something like vLLM and keep retrieval, logs, and audit metadata inside our environment."

⚙ AI Development Practices

Two sides of the same coin: building AI products AND building any product with AI as your co-engineer. Both require discipline. Both are becoming table stakes.

Part A: Building Products WITH AI

Using AI coding assistants, agents, and orchestration to build any software, not just AI products.

1. AI as Engineering Partner, Not Magic Wand

The mindset shift

AI does not replace engineering judgment. It accelerates it. The engineer still owns: architecture, tradeoffs, security, correctness, and the final product. AI handles: scaffolding, boilerplate, exploration, drafting, and tedious translation work.

The difference between a junior using AI and a principal using AI: the junior accepts the output. The principal directs the process, constrains the scope, verifies the result, and knows when the AI is confidently wrong.

Good AI-assisted engineering
  • Describe the problem with full context
  • Specify inputs, outputs, constraints
  • Define acceptance criteria upfront
  • Review generated code for security
  • Verify behavior, do not assume correctness
  • Use AI to explore options, then YOU decide
Bad AI-assisted engineering
  • "Build me a thing" with no constraints
  • Accept output without reading it
  • Trust generated auth/security code blindly
  • No tests, "it seemed to work"
  • Let AI decide architecture for you
  • Skip review because "AI wrote it"
"I use AI as an accelerator, not as an unreviewed source of truth. I still verify architecture, security implications, edge cases, and failure handling myself. The AI handles implementation speed. I handle engineering judgment."

2. Spec-Driven Development

The single most impactful practice for building with AI: write the spec before writing (or generating) any code.

Why specs matter more with AI High impact

Without a spec, AI will happily generate code that works but is architecturally wrong, insecure, or misaligned with the rest of the system. A spec constrains the AI to produce what you actually need.

A good spec for an AI coding session includes:

  • Goal: what problem this code solves and why
  • Inputs and outputs: exact data shapes, types, formats
  • Interfaces: what APIs does it consume or expose? What contracts must it respect?
  • Constraints: what it must NOT do, what it must NOT touch
  • Security requirements: auth, validation, secrets handling
  • Acceptance criteria: how do we know it works?
  • Integration context: what other services/components exist, and how does this one connect?
A spec is like a blueprint for a contractor. Without it, a talented contractor might build you a beautiful room, but it is in the wrong part of the house, uses the wrong plumbing standard, and has no door to the hallway. The spec ensures the AI builds the right thing in the right place with the right connections.

3. The Orchestrator Pattern

For non-trivial projects, the most effective AI-assisted workflow is not one long conversation. It is structured delegation.

How it works in practice

You have one main conversation (the orchestrator) that understands the full system. It does not write all the code. Instead, it:

  1. Understands the full product by discussing requirements, users, constraints, and scope with you
  2. Designs the architecture by exploring options, tradeoffs, and making decisions together
  3. Defines the service map by identifying what components/services the system needs
  4. Creates spec sheets for each agent, one per service or concern area:
    • Frontend spec: UI requirements, component list, state management, responsiveness
    • Backend/API spec: endpoints, request/response shapes, auth, business logic
    • Database spec: schema, migrations, indexing, relationships, retention
    • Security spec: auth model, encryption, audit requirements, RBAC rules
    • AI/ML spec: model selection, RAG config, prompt templates, eval criteria
    • Infrastructure spec: deployment, CI/CD, monitoring, scaling
  5. Manages execution by assigning work to specialized agents, tracking progress, and resolving conflicts
  6. Verifies integration by checking that all pieces actually connect and work together
What each subagent spec should contain

Each agent has ZERO context from the orchestrator conversation. Its spec is its entire world. Include:

  • Goal: what this agent is building and why it exists in the system
  • Scope: exactly which files/services it owns
  • Interfaces: what it consumes from other services, what it exposes
  • Shared contracts: API shapes, data models, error formats that ALL agents must respect
  • What other agents are building: enough context to understand how this piece fits
  • Constraints: what NOT to modify, what patterns to follow, security requirements
  • Tech stack: languages, frameworks, libraries to use
  • Acceptance criteria: specific, testable conditions for "done"
The #1 failure with multi-agent development: agents build things that do not connect. The frontend expects one API shape, the backend builds another. This happens because the orchestrator did not define shared contracts explicitly. Always define and distribute the exact interface contracts before any agent starts building.

4. Git Isolation for AI Work

Git worktrees / feature branches Best practice

Each AI agent or session works on its own isolated git branch or worktree. This gives you:

  • Safety: if an agent produces garbage, main is untouched
  • Parallelism: multiple agents can work simultaneously without conflicts
  • Review gate: the orchestrator reviews and merges, agents never merge directly
  • Clean history: each feature or service gets a clean commit history on its branch

The orchestrator (or a dedicated integration role) is the only one who merges into main, after verifying integration.

5. Testing with AI Assistance

Test generation

AI is very good at generating tests, and this is one of its highest-value uses. The practice:

  • Write the implementation (or have AI write it)
  • Ask AI to generate tests for the implementation
  • Review the tests: are edge cases covered? Are assertions meaningful?
  • Add adversarial tests: what should break? What inputs should be rejected?

Better yet: write the tests FIRST (TDD), then have AI implement to pass them. This gives you more control over correctness.

Integration testing in multi-agent workflows

When multiple agents build separate services, integration testing is critical. Practices:

  • Contract tests: verify that each service's actual API matches the shared contract spec
  • Smoke tests on merge: every merge to main triggers basic end-to-end checks
  • Dedicated integration agent: a separate agent (or the orchestrator) that only runs integration tests across services as they complete
  • Automated on CI: pushes and merges trigger integration test suites automatically

6. Security When AI Writes Your Code

AI-generated code has specific security risks that human-written code does not.

Known risks Must address
  • Hallucinated packages: AI sometimes suggests npm/pip packages that do not exist. Attackers register those names and publish malicious packages. Always verify dependencies exist and are legitimate.
  • Insecure defaults: AI often generates code that works but uses weak patterns: hardcoded secrets, disabled SSL verification, overly permissive CORS, SQL concatenation instead of parameterized queries.
  • Copy-paste vulnerabilities: AI may reproduce vulnerable patterns from its training data, including known CVE-affected code patterns.
  • Over-privileged code: AI tends to request broad permissions (admin access, wildcard IAM policies) because it optimizes for "works" over "secure."
  • Missing validation: AI often generates the happy path well but skips input validation, error handling, and boundary checks.
Defensive practices
  • Constrain the AI upfront: include security requirements in every prompt. "Do not hardcode secrets. Use parameterized queries. Validate all input. Use least-privilege permissions."
  • Review auth code manually: never trust AI-generated authentication, authorization, or encryption code without careful human review
  • Dependency verification: check that every suggested package exists, is maintained, and is not compromised
  • Static analysis: run linters and security scanners (Semgrep, Snyk, CodeQL) on AI-generated code
  • Never commit secrets: AI sometimes generates example API keys or credentials. Always check before committing.
  • Destructive operations: review any AI-generated code that deletes data, modifies permissions, runs migrations, or touches infrastructure with extra caution
"When AI writes code, I treat it like a talented but careless junior developer's pull request. I trust the structure but verify the details, especially around auth, secrets, permissions, input validation, and destructive operations. I include security constraints in every prompt, run static analysis on generated code, and never let AI-generated auth or encryption code ship without manual review."

7. Architecture Decision-Making with AI

How to use AI as an architecture partner

AI is excellent at widening your option space and surfacing tradeoffs you might miss. The practice:

  1. Describe the problem and constraints: users, scale, compliance needs, team size, timeline
  2. Ask for options: "What are 3 different architectures for this? What are the tradeoffs of each?"
  3. Challenge each option: "What are the failure modes? What happens at 10x scale? What is hardest to change later?"
  4. Ask for critique: "What did I miss? What would a staff engineer challenge here? What are the hidden risks?"
  5. Make the decision yourself: AI informs, you decide. Document the decision in an ADR.
Using AI to learn and onramp fast

When entering a new domain or technology:

  1. Ask for layered explanation: high level first, then architecture, then implementation details
  2. Ask for comparisons: "How does X compare to Y? When would I choose one over the other?"
  3. Ask for decision criteria: "When is X overkill? What are the signs X is wrong for my case?"
  4. Ask for artifacts: "Draft an ADR for this decision. Create a threat model. Write the API contract."
  5. Ask for adversarial review: "Critique this design. What breaks? What did I forget?"

This is how you use AI to compress weeks of research into hours, without outsourcing judgment.

The pattern: AI explores the space, you make the decisions, documentation captures the reasoning. This is faster than solo research but more rigorous than "just ask AI to build it."

9. Living Documentation

AI systems have more moving parts than traditional software. If you do not document what is running, what version, and why, you will lose track fast.

Architecture Doc Non-negotiable

A running document that shows: system components, data flow, model versions, retrieval strategy, tool configurations, trust boundaries. Updated with every significant change, not written once and forgotten.

This is the single most important document in an AI project. When something breaks at 2am, this is what the oncall engineer reads.

Architecture Decision Records (ADRs)

Short docs for each important decision. "Why RAG instead of fine-tuning." "Why pgvector instead of Pinecone." "Why we added a human approval gate for contract generation."

These are gold six months later when someone asks "why did we do it that way?"

AI Governance Sheet

For each AI component: what model, what data it sees, what it may and may not do, human review points, evaluation criteria, safety controls, logging expectations. Think of it as the "rules of engagement" for each AI-powered feature.

Runbooks

What to do when the AI does something wrong. How to rollback a prompt change. How to investigate a bad retrieval. How to handle a customer complaint about AI output. These should exist before you need them.

The pattern that works: treat documentation as a continuous practice, not a phase. Every PR that changes AI behavior should update the relevant doc. Tools like GSD (Get Stuff Done) enforce this by requiring documentation updates as part of the workflow before marking tasks complete.

10. Eval-Driven Development

This is the AI equivalent of test-driven development, and it is becoming the defining practice of serious AI teams.

The core idea

Before you change a prompt, retrieval strategy, or model version, define how you will measure whether it got better or worse. Then make the change. Then measure.

Without evals, you are flying blind. "It seemed better when I tried it" is not engineering.

What to evaluate
  • Retrieval quality: Are the right chunks coming back? Relevance, precision, recall.
  • Answer faithfulness: Is the answer actually supported by the retrieved sources?
  • Task correctness: For extraction, did it get the fields right? For classification, is the label correct?
  • Safety: Does it refuse when it should? Does it leak information it should not?
  • Latency and cost: How long and how expensive per request?
Golden datasets

A curated set of question/answer pairs where you know the correct answer. Run your system against these before every change. If scores drop, the change does not ship.

Production failures should be captured and added to the golden dataset over time, turning real incidents into permanent regression tests.

Eval-driven development is like a chef tasting the dish at every step. You do not just throw ingredients in and serve it. You taste after each addition, compare to the standard, and adjust. The golden dataset is your reference palate.

11. Prompt Versioning & Management

Prompts are code. They affect behavior. They need the same discipline as code changes.

Why this matters

A single word change in a system prompt can completely alter output quality, safety, and reliability. Without versioning, you cannot:

  • Know which prompt version produced a specific output
  • Roll back when a change degrades quality
  • Compare performance across versions
  • Audit what instructions were active when an incident occurred
Best practices
  • Version every change: track author, timestamp, diff, and intent
  • Staged deployment: dev, staging, production, just like code
  • Eval before promotion: run golden dataset before promoting to production
  • Rollback path: one-click revert to the last known good version
  • Documentation with each version: why was this changed? What was it supposed to improve?

12. AI Observability & Tooling

You cannot improve what you cannot see. AI observability is the practice of monitoring, tracing, and analyzing AI systems to understand behavior and detect issues.

What to observe in AI systems
  • Traces: full request path: user input, prompt template, retrieved docs, model call, tool calls, output
  • Latency: per-step and end-to-end response time
  • Token usage / cost: per request and aggregate
  • Retrieval quality: what chunks were retrieved and how relevant were they
  • Eval scores: automated quality checks on production responses
  • Failure/fallback rates: how often does the system fail or degrade gracefully
  • Human override rate: how often do humans correct or reject AI output

13. Security Practices Developers Are Talking About

AI security is rapidly evolving. These are the practices and standards that matter right now.

OWASP LLM Top 10 (2025) Must know

The most referenced security framework for LLM applications. Updated in 2025 with significant changes:

  1. Prompt Injection - still #1. Found in 70% of security audits. Now includes multimodal attacks (malicious instructions in images, QR codes).
  2. Sensitive Information Disclosure - model leaking PII, credentials, or internal data
  3. Supply Chain Vulnerabilities - compromised models, plugins, training data, dependencies
  4. Data and Model Poisoning - corrupted training data causing harmful behavior
  5. Improper Output Handling - trusting model output without validation before downstream use
  6. Excessive Agency - expanded for agentic AI. Unchecked permissions, tools without boundaries
  7. System Prompt Leakage - NEW. Sensitive info in system prompts exposed to users
  8. Vector and Embedding Weaknesses - NEW. RAG-specific: poisoned embeddings, malicious data injection into retrieval
  9. Misinformation - model confidently generating false information
  10. Unbounded Consumption - expanded from DoS. Resource exhaustion, unexpected cost explosions
Red Teaming / Adversarial Testing Growing fast

Deliberately trying to break your AI system before attackers do. Two approaches:

  • Manual red teaming: human testers craft creative attacks. Better at finding subtle, nuanced failures.
  • Automated red teaming: tools that generate thousands of adversarial prompts. Better for broad coverage.

Key statistics: multi-turn jailbreaks succeed 97% of the time within 5 conversational turns. Role-play attacks succeed nearly 90% of the time. This is why guardrails must live outside the model.

Red teaming tools to know

Promptfoo
Open-source. Tests prompts, agents, and RAG for injections, jailbreaks, data leaks.
Garak
NVIDIA's open-source LLM vulnerability scanner. Probes for known weakness patterns.
DeepTeam
By Confident AI. Red teaming framework with automated attack generation.
Guardrails Architecture Critical pattern

The most important security architecture principle for AI systems:

Guardrails must live OUTSIDE the model, not inside the prompt.

You cannot rely on "please don't do bad things" in a system prompt. Prompt injection can bypass it. Instead:

  • Input guardrails: validate/sanitize user input before it reaches the model
  • Output guardrails: validate model output before it reaches users or downstream systems
  • Tool guardrails: allowlisted tools only, permission checks in code, not in prompts
  • Action guardrails: human approval for destructive/irreversible actions
  • Cost guardrails: rate limits, token budgets, circuit breakers
Meta's "Rule of Two" (published October 2025): for any dangerous tool call, at least two independent safety checks must pass before execution. File-type firewalls, human approvals, and kill switches for tool calls cannot depend on model behavior alone.
Shift-Left Security for AI

"Shift left" means addressing security from the design phase, not after deployment. For AI systems this means:

  • Threat model the AI components during architecture design
  • Define trust boundaries between model, tools, retrieval, and users
  • Include adversarial test cases in your eval suite from day one
  • Review prompt templates for information leakage before deployment
  • Classify data sensitivity before it enters the RAG pipeline
  • Audit model/embedding supply chain (where did this model come from? who trained it?)

14. Agent Safety Practices

As AI agents gain more autonomy and tool access, safety practices are evolving fast.

The principle

An agent should have the minimum authority needed for its task, verified at every step, with hard boundaries that code enforces regardless of what the model says.

Practical controls
  • Tool allowlists: model can only call pre-approved functions
  • Parameter validation: tool call arguments are validated before execution
  • Bounded iterations: max loop count, max token budget, max time
  • Approval gates: pause for human review before high-risk actions
  • Kill switches: ability to halt an agent immediately
  • Sandboxing: agents run in isolated environments with limited access
  • Audit trail: every tool call, every decision point, every state change is logged

Foundation Models

The model behind ChatGPT. Accessed via OpenAI's API. You send a prompt, you get a response. Pricing is per-token (per unit of text processed).

Meta's open-weight model. You can download it and run it on your own hardware. This is the foundation for most self-hosted enterprise LLM deployments.

French AI company. Open-weight models known for strong performance relative to their size. Popular for self-hosted deployments where you want good quality at lower compute cost.

Open-weight models (LLaMA, Mistral) give you full control but you own the infrastructure, ops, and security. Agent / Orchestration Frameworks

These are code libraries for building multi-step AI workflows. They are not models. They coordinate how models, tools, and data work together.

LangChain Most widely known

What it is: a Python/JS library with pre-built connectors for LLMs, document loaders, vector stores, prompt templates, and output parsers.

What it actually does: saves you from writing boilerplate. Instead of writing your own "load PDF, chunk it, embed it, store it, retrieve it, prompt the LLM" code, LangChain provides modules for each step.

When to use: prototyping RAG pipelines quickly. Connecting multiple components without writing every integration yourself.

Honest assessment: great for getting started. Many production teams eventually replace it with direct API calls because the abstractions can make debugging harder at scale. It is a toolkit, not a runtime.

LangGraph

What it is: a separate library from the LangChain team for building stateful, multi-step agent workflows as graphs (nodes + edges + shared state).

What it actually does: lets you define complex workflows where an AI can loop back, branch, use tools, pause for human approval, and maintain state across steps. Built-in checkpointing so workflows survive crashes.

When to use: complex agentic systems with branching logic, retries, human-in-the-loop, multi-agent coordination. Overkill for simple RAG.

OpenAgents

What it is: an open-source research platform for building and hosting AI agents that can browse the web, write code, and use tools.

Interview context: demonstrates familiarity with the broader agent ecosystem. Less commonly used in production than LangChain. Know that it exists and what role it fills.

AutoGPT

What it is: one of the first "autonomous agent" projects. Gives an LLM a goal and lets it recursively plan and execute steps to achieve it, using tools and web access.

Interview context: historically important as a demonstration of agentic AI. In practice, fully autonomous agents are unreliable for enterprise use. Know the concept, know why it is risky (unbounded loops, hallucinated plans, no human oversight), and know that production systems need more control.

"I'm familiar with the agent framework landscape. LangChain is useful for quickly wiring together LLM, retrieval, and tool-use components. LangGraph adds stateful orchestration with branching and human-in-the-loop for complex workflows. For production, I'd evaluate whether the framework's abstractions help or hinder debuggability, and I'd always prefer deterministic workflow control where possible over fully autonomous agent behavior."

MLOps / Lifecycle Tools

These manage the operational side of ML: tracking experiments, orchestrating data pipelines, and running training/deployment workflows.

MLflow

What it is: an open-source platform for managing the ML lifecycle.

Plain English: imagine you train 50 different models with different settings. MLflow records every run: what parameters you used, what data you trained on, what scores you got, and which model artifact came out. It is a lab notebook for ML experiments. It also provides a model registry where you can promote models from "experimental" to "staging" to "production."

Key capabilities: experiment tracking, model versioning, model registry, run comparison, artifact storage.

Airflow (Apache Airflow)

What it is: a workflow orchestration platform. Not AI-specific. Used broadly for data engineering.

Plain English: you define a pipeline as a DAG (Directed Acyclic Graph): "first extract data from the database, then clean it, then generate features, then train the model, then evaluate." Airflow runs these steps in order, on schedule, with retry logic if a step fails. Think of it as a sophisticated cron job system for data pipelines.

Key capabilities: scheduled workflows, dependency management, retry/alerting, integrations with databases/cloud/APIs.

Kubeflow

What it is: an ML platform that runs on Kubernetes.

Plain English: if your organization runs on Kubernetes and wants to do ML training and deployment as containerized workloads, Kubeflow provides the glue. It can orchestrate training jobs, hyperparameter tuning, model serving, and notebook environments, all as Kubernetes resources.

Key capabilities: ML pipelines on K8s, distributed training, model serving, notebook servers.

When to use: organizations already invested in Kubernetes who want ML infrastructure integrated with their existing platform.

Airflow is like a factory floor manager: it decides what job runs when and in what order. MLflow is like a quality control logbook: it records what happened in each production run and tracks which product version is approved. Kubeflow is the factory itself: the physical infrastructure where the work happens, running on Kubernetes containers.

Cloud AI Platforms

Managed platforms inside the big cloud providers. They bundle model access, training infrastructure, deployment, and governance into one service.

AWS SageMaker

What it is: Amazon's managed ML platform on AWS.

What it does: provides managed notebooks for development, training infrastructure that provisions and tears down automatically, model hosting endpoints, built-in algorithms, hyperparameter tuning, and a model registry.

When to use: you are on AWS and want managed ML infrastructure without building your own training/serving platform.

Google Vertex AI

What it is: Google Cloud's unified AI/ML platform.

What it does: access to Gemini models, custom model training on Google TPUs/GPUs, model evaluation, agent tooling, and RAG grounding services. Integrates with GCP IAM, storage, and networking.

When to use: you are on Google Cloud. Especially strong for Gemini-based applications and for orgs wanting managed GenAI + classic ML in one platform.

Azure ML / Microsoft Foundry

What it is: Microsoft's AI platform on Azure. Azure ML handles classic ML lifecycle (training, compute, pipelines, endpoints). Microsoft Foundry is the newer unified platform for GenAI apps, agents, and governance.

When to use: you are on Azure. Foundry for GenAI app/agent development with governance. Azure ML for classic ML training and deployment pipelines.

You do not need deep expertise in all three platforms. The interview signal is: "I understand that each cloud has a managed AI platform, I know roughly what they provide, and I would choose based on which cloud the organization already uses." That is the right level.

ML Libraries

These are the code libraries you would actually write training and inference code with. Different tools for different jobs.

Scikit-learn Learn first for this role

What it is: a Python library for classic machine learning on structured/tabular data.

Plain English: if your data lives in spreadsheets or database tables (customers, transactions, orders, tickets), and you want to predict something (churn, fraud, revenue), scikit-learn is usually the first tool you reach for. It has all the standard algorithms: linear regression, logistic regression, decision trees, random forests, support vector machines, clustering.

Why it matters for this role: the JD includes forecasting and predictive analytics. That is often scikit-learn territory, not LLM territory.

PyTorch

What it is: a deep learning framework. The most popular one for both research and industry.

Plain English: when you need neural networks (for text, images, audio, or complex patterns that simpler models cannot capture), PyTorch is how you build and train them. Most modern LLMs and embedding models are built with PyTorch.

For this role: you probably will not be training neural networks from scratch. But understanding that PyTorch powers the models you use (and knowing it exists) is important vocabulary.

TensorFlow

What it is: Google's deep learning framework. Functionally similar to PyTorch but with different design philosophy.

Plain English: does the same job as PyTorch. Was dominant earlier, PyTorch has become more popular in recent years. Still widely used, especially in Google ecosystem and production deployment (TensorFlow Serving, TFLite for mobile).

For this role: same as PyTorch. Know it exists, know what it does, know it is the other major deep learning option.

Hugging Face Very important for this role

What it is: an open platform and library for transformer models. Think of it as the "GitHub + npm of ML models."

What it actually provides:

  • Model Hub: thousands of pre-trained models you can download and use (LLMs, embedding models, classifiers, image models)
  • Transformers library: Python code to load, run, and fine-tune these models
  • Tokenizers: fast text processing tools that convert text to the format models expect
  • Datasets: curated training and evaluation datasets

Why it matters: when someone says "use a Hugging Face model," they mean "download a pre-trained model from the Hub and use it via the transformers library." This is how you would get an embedding model for your RAG pipeline, or a base model for fine-tuning, without using a paid API.

Retrieval / Vector Search

Tools for storing embeddings and searching by meaning. The retrieval backbone of RAG systems.

FAISS (Facebook AI Similarity Search) Open source

What it is: a library (not a database) for fast similarity search over vectors.

Plain English: you give it a million vectors, it builds an index, and then you can query "find the 10 closest vectors to this one" in milliseconds. It runs in-memory and is extremely fast.

Tradeoff: it is a building block, not a full database. No built-in metadata filtering, no access control, no persistence management out of the box. You build on top of it.

When to use: you want maximum speed and are willing to handle persistence and filtering yourself. Good for embedding into a larger system.

Pinecone Managed service

What it is: a fully managed vector database as a cloud service.

Plain English: you upload vectors via API, Pinecone stores and indexes them, you query via API and get results back. No infrastructure to manage. Includes metadata filtering, namespaces, and scaling.

Tradeoff: data goes to Pinecone's cloud. Cost scales with usage. Vendor dependency.

When to use: you want fast time-to-market without managing vector infrastructure. Good for prototypes and teams without dedicated infrastructure engineers.

Others worth knowing
  • pgvector: Postgres extension. Add vector search to your existing Postgres. Great when you want one database for structured data AND vectors.
  • Weaviate: open-source vector database with built-in vectorization, filtering, and hybrid search.
  • Milvus: open-source, built for very large scale. Popular in enterprise deployments.
  • Chroma: lightweight, developer-friendly. Good for prototyping.
"For vector search, I'd choose based on scale and operational constraints. pgvector is great when you already use Postgres and have moderate scale, since it keeps everything in one database. FAISS gives you raw speed as a library you embed into your application. Pinecone is fastest to start with as a managed service. For large-scale self-hosted enterprise deployments, I'd evaluate Milvus or Weaviate."

Compliance & Security Frameworks

Not software. These are standards and certifications that govern how systems must be built and operated in sensitive environments. This role explicitly cares about these.

NIST (National Institute of Standards and Technology) Must know

What it is: a US government agency that publishes standards and frameworks for cybersecurity, AI risk management, and software development.

Key frameworks to know by name:

  • NIST Cybersecurity Framework (CSF): the standard framework for managing cybersecurity risk. Identify, Protect, Detect, Respond, Recover.
  • NIST AI Risk Management Framework (AI RMF): guidance for trustworthy AI design, development, and deployment. Covers bias, safety, explainability, privacy.
  • NIST SSDF: Secure Software Development Framework. Practices for secure software development lifecycle.

For this role: when the JD says "NIST," they want you to be aware that these frameworks exist and that you design systems with their principles in mind, not that you memorized every control number.

CMMC (Cybersecurity Maturity Model Certification)

What it is: a DoD (Department of Defense) certification framework that requires defense contractors to meet specific cybersecurity practices based on the sensitivity of the data they handle.

Plain English: if your company does work for the US military or defense agencies, you must prove your cybersecurity meets certain levels. CMMC has multiple levels. Level 1 is basic hygiene. Level 2 requires implementing 110 practices from NIST SP 800-171. Level 3 is for the most sensitive work.

For this role: Day & Zimmermann does defense/government work. This is why CMMC is in the JD. You do not need to know every control, but you should understand that it constrains how you build, host, and secure AI systems.

FedRAMP (Federal Risk and Authorization Management Program)

What it is: a standardized approach for security assessment and authorization of cloud products used by US federal agencies.

Plain English: if you want the federal government to use your cloud-hosted software, it needs to pass FedRAMP authorization. This means extensive security controls, continuous monitoring, and formal assessment by third-party auditors.

For this role: if the company is building AI systems that touch government data or run in government-approved cloud environments, FedRAMP compliance affects which cloud services, hosting options, and tools you can use.

DoD (Department of Defense)

Not a framework itself, but the umbrella context. DoD standards generally mean: stricter data handling, classified information protocols, CMMC compliance, FedRAMP-authorized cloud environments, and heightened security review for any AI/ML system processing defense-related data.

"I understand that NIST provides the foundational cybersecurity and AI risk management frameworks, CMMC is the defense contractor certification that enforces specific cybersecurity maturity levels, and FedRAMP governs cloud service authorization for federal use. For AI systems in these environments, this means stricter data residency, access controls, auditability, and often self-hosted or FedRAMP-authorized infrastructure rather than standard commercial cloud APIs."

The 8 Reusable Answer Templates

Template 1: "How would you build X?"
"First I'd clarify the exact task and success criteria. Then I'd identify the input data, output format, and operational constraints like latency, cost, and sensitivity. From there I'd choose the simplest architecture that fits. Then I'd define failure modes, evaluation metrics, deployment strategy, and monitoring before calling it production-ready."
Template 2: "Why RAG vs fine-tuning?"
"I use RAG when the challenge is access to changing external knowledge and the need for traceability. I use fine-tuning when I need the model to learn a stable behavior or domain-specific response pattern that prompting alone can't reliably achieve. If the issue is knowledge freshness, fine-tuning is usually the wrong first move."
Template 3: "When should you use an agent?"
"I'd use an agent only when the problem truly benefits from multi-step decision-making, tool use, and dynamic branching. If the workflow is predictable and fixed, I'd rather build a deterministic pipeline because it's easier to test, cheaper, and safer."
Template 4: "How do you evaluate an LLM system?"
"I evaluate at the component level and the end-to-end level. For RAG, I look at retrieval quality and faithfulness separately from final answer quality. For extraction, I focus on field accuracy. For production, I also track latency, cost, fallback behavior, and human override rates. I want both offline evals and real usage signals."
Template 5: "How do you secure it?"
"I start with data classification and least privilege. Then I ensure retrieval is ACL-aware, tools are permission-bounded, prompts and retrieved content are treated as untrusted inputs, actions are policy-validated outside the model, and all important interactions are auditable. High-risk outputs should have human review gates."
Template 6: "What are the failure modes?"
"I'd break failure modes into retrieval problems, reasoning/generation problems, orchestration problems, and governance problems. That gives a more complete view than focusing only on hallucinations."
Template 7: "How would you measure business impact?"
"I'd define a baseline first, then measure improvement in the business metric that actually matters: turnaround time, analyst hours saved, accuracy, error rate, forecast quality, or win rates, depending on the use case. I'd avoid declaring success based only on demo quality."
Template 8: "What if you don't know the exact tool?"
"I haven't used that one heavily in production, but I understand the role it plays in the stack. In systems I've worked on, I've approached that layer through [adjacent approach], so I'd evaluate it based on fit, operational complexity, observability, and security requirements."

How to Describe Your AI-Driven Engineering Workflow

When they ask "How would you use AI to build a system?", here is a clean answer:

"I start by using AI to clarify the problem space: user goals, inputs, outputs, constraints, and scope boundaries. Then I explore architecture options and tradeoffs interactively, aiming for the simplest design that can deliver MVP value without painting us into a corner. Early on I lean toward a modular monolith over premature microservices, while keeping interfaces clean enough to extract later.

Once architecture is chosen, I use AI to generate structured artifacts: requirements, design docs, API contracts, security considerations, and rollout plans. For implementation, I like specialized workstreams with clear ownership boundaries, but with a central orchestration layer for integration and consistency.

CI validates changes continuously. CD packages versioned artifacts and promotes them through environments safely. In production, I pair that with monitoring, structured logs, traces, and audit events. For AI-specific functionality, I'm deliberate about where the model assists versus where deterministic code must own permissions, business rules, and irreversible actions."
